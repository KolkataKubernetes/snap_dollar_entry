## Quick Summary

**Goal**

Port the legacy SNAP dollar-entry workflow into the new directory structure under `1_code/`, while making script paths consistent, outputs reproducible, and validation explicit. The refactor matters only if the new pipeline can reproduce the core descriptive and reduced-form results currently generated by `legacy/Box/code/02 - Descriptives & Motivation US.R`.

**Deliverable**

A working, file-by-file R pipeline in the main repository that replaces the legacy monolithic scripts with standardized ingest, descriptives, and reduced-form scripts. The deliverable includes processed intermediate files written via the root-path text files and a reproducible validation routine that compares new outputs against the legacy analysis target.

**Success Criteria**

- The scripts in `1_code/1_0_ingest/` run without hardcoded absolute data paths inside the R code.
- The monolithic legacy U.S. analysis is decomposed into scripts under `1_code/1_1_descriptives/` and `1_code/1_2_reduced_form/`.
- Running the new pipeline produces the key processed datasets, descriptive artifacts, and reduced-form estimates needed to match the legacy U.S. analysis.
- The new outputs are close enough to the legacy benchmark that any remaining differences can be explained line-by-line from documented cleaning or aggregation changes.

## Purpose / Big Picture

After this refactor, a contributor should be able to start from the main repository layout, run the ingest scripts to build the needed county-year datasets, run a small set of descriptive scripts to recreate the workshop-style visuals, and run a small set of reduced-form scripts to recreate the U.S. analysis. The user-visible improvement is that the workflow is no longer trapped inside `legacy/` or one oversized script with mixed responsibilities, mixed path conventions, and implicit intermediate files.

The benchmark is behavioral, not cosmetic. The refactor succeeds only if the new workflow reproduces the analytical content of `legacy/Box/code/02 - Descriptives & Motivation US.R`, especially the county-year treatment panel, the retailer stock descriptives, and the event-study estimates.

## Context and Orientation

The main repository now has a new top-level structure with `0_inputs/`, `1_code/`, and `2_processed_data/`. The intended destination for the refactor is `1_code/`, which currently contains only four ingest scripts in `1_code/1_0_ingest/`:

- `1_code/1_0_ingest/1_0_0_SNAP_waiver_ingest.R`
- `1_code/1_0_ingest/1_0_1_SNAP_retailer_ingest.R`
- `1_code/1_0_ingest/1_0_2_unemployment_rates.R`
- `1_code/1_0_ingest/1_0_3_ACS_prep.R`

The folders `1_code/1_1_descriptives/` and `1_code/1_2_reduced_form/` exist but are empty. This means the current new layout is only a partial scaffold.

The legacy logic is split across two places. The older intermediate scripts are in `legacy/1_code/`, including `legacy/1_code/1_0_0_SNAP_waiver_ingest.R` and `legacy/1_code/1_0_2_add_covariates.R`. The more complete U.S. analysis scripts are in `legacy/Box/code/`, especially:

- `legacy/Box/code/00 - Cleaning SNAP.R`
- `legacy/Box/code/00 - Unemployment Rates.R`
- `legacy/Box/code/00 - Prepping ACS .R`
- `legacy/Box/code/02 - Descriptives & Motivation US.R`

In this repository, “relative pathing regime” does not mean storing all data in Git. It means that an R script should compute its input and output locations by reading repository-local pointer files such as `0_inputs/input_root.txt` and `2_processed_data/processed_root.txt`, rather than embedding machine-specific absolute paths inside the script source.

The current pointer files contain external Box-backed paths:

- `0_inputs/input_root.txt` points to the raw-input root.
- `2_processed_data/processed_root.txt` points to the processed-data root.

This is acceptable for now, but every main-repo script must read those pointer files consistently. The scripts must not mix three styles at once: hardcoded absolute paths, relative repo paths, and old `Data/...` references.

## Data Artifact Flow

The refactor should make the artifact flow explicit and reproducible.

Raw Inputs

- Raw waiver panel files under the root pointed to by `0_inputs/input_root.txt`
- SNAP retailer locator raw file used by `legacy/Box/code/00 - Cleaning SNAP.R`
- County crosswalk input used to attach county FIPS
- ACS source files used by `legacy/Box/code/00 - Prepping ACS .R`
- Unemployment source files used by `legacy/Box/code/00 - Unemployment Rates.R`
- Rural-urban continuum file used by the descriptives section in `legacy/Box/code/02 - Descriptives & Motivation US.R`

Intermediate Artifacts

- Consolidated waiver panel written by `1_code/1_0_ingest/1_0_0_SNAP_waiver_ingest.R`
- Clean SNAP retailer file written by `1_code/1_0_ingest/1_0_1_SNAP_retailer_ingest.R`

Final Outputs

- Descriptive figures under a stable output directory defined by the new scripts
- Regression tables or serialized model outputs under a stable output directory defined by the new scripts
- A validation note or machine-readable comparison artifact showing agreement with the legacy U.S. script

## Plan of Work

First, normalize the ingest layer. `1_code/1_0_ingest/1_0_0_SNAP_waiver_ingest.R`, `1_code/1_0_ingest/1_0_1_SNAP_retailer_ingest.R`, `1_code/1_0_ingest/1_0_2_unemployment_rates.R`, and `1_code/1_0_ingest/1_0_3_ACS_prep.R` must all use the same preamble style, the same numbered section style, and the same path-loader helpers. Each script must declare its inputs, procedures, and outputs in the preamble, and each must read from the root pointer files instead of embedding absolute paths. While doing this, preserve the legacy transformation logic from the matching `legacy/Box/code/00 - ...` scripts or `legacy/1_code/` files, but rewrite the code in tidyverse-forward style when that does not change behavior.

Second, create a single script that assembles the analysis-ready county-year panel. The legacy U.S. script currently does this assembly inline before making figures and running regressions. In the new layout, that assembly should become an explicit intermediate step that merges waiver treatment, store outcomes, wages, ACS controls, rent/income measures, population, and unemployment into one processed file. This keeps descriptives and reduced-form scripts thin and easier to validate.

Third, split the descriptive portion of `legacy/Box/code/02 - Descriptives & Motivation US.R` into separate scripts under `1_code/1_1_descriptives/`. Each script should produce exactly one figure or one closely related figure family and should read the analysis-ready processed panel rather than rebuilding data objects internally. Based on the current legacy section, the initial descriptive split should include the retailer format stock index, the rural-only retailer format stock index, the county-waiver time series, the pre/post retail-format growth chart, and the dollar-store stock trend by waiver status.

Fourth, split the reduced-form portion of `legacy/Box/code/02 - Descriptives & Motivation US.R` into scripts under `1_code/1_2_reduced_form/`. At minimum, one script should build the analysis sample and event-time variables, and another should run the event-study models and write tables or model summaries. If the regression code remains tightly coupled, keep it in one file initially but make the output contract explicit.

Fifth, define and run validation. Validation should operate at three levels: input/output checks for ingest scripts, panel-contract checks for the assembled analysis dataset, and benchmark checks against the legacy U.S. script. The benchmark cannot remain vague. Before execution is complete, this plan must list the exact figures, table outputs, or coefficient sets that count as “matching.”

## Validation and Acceptance

This refactor changes executable analytical logic, so validation must be reproducible and staged.

Validation check 1: path-regime correctness for ingest scripts.

Run each ingest script with `Rscript` from the repository root after temporarily moving to a machine where the Box data root is reachable only through the text pointer files, not through hardcoded paths embedded in source. The test passes only if the scripts complete and write outputs using the configured roots.

Validation check 2: analysis-panel contract.

After building the new analysis-ready county-year panel, verify that it contains the required keys and core columns. At minimum, the panel must contain one row per county-year and include county identifier, year, treatment indicator, event year, dollar-store outcome measures, and control variables used in the event study. This test should fail before the panel-builder script exists and pass after it is implemented.

Validation check 3: benchmark equivalence with the legacy U.S. script.

Run `legacy/Box/code/02 - Descriptives & Motivation US.R` and the new split pipeline against the same data roots. Compare:

- the count of treated counties
- the year range in the final panel
- at least one descriptive artifact reproduced from the legacy script
- at least one reduced-form table or coefficient vector reproduced from the legacy script

The exact comparison target must be upgraded from this placeholder list to named files or named coefficient sets before the plan moves to `Execution (In Progress)`.

## Idempotence and Recovery

Each ingest script should be safe to rerun. If an output file already exists, the script should either overwrite it deterministically or write a versioned replacement according to one documented rule. The scripts should not append duplicate rows across reruns.

The analysis-panel build should also be idempotent. If a rerun fails midway, recovery should consist of deleting only the incomplete output artifact and rerunning the script, not manually editing intermediate data.

If legacy and new outputs differ, do not “fix” the difference by hand-editing processed files. Instead, record the discrepancy, trace it to the upstream script or data contract, and update either the implementation or the plan.

## Data Contracts, Inputs, and Dependencies

Dependency 1: root path pointer files.

- Tool: base R `readLines()`
- Repository location: `0_inputs/input_root.txt`, `2_processed_data/processed_root.txt`
- Inputs consumed: single-line text files pointing to the raw-input root and processed-data root
- Outputs produced: normalized directory strings used by downstream file builders
- Invariant: no main-repo script may require editing its source code to switch machines if the pointer files are updated correctly

Dependency 2: waiver ingest.

- Tool: R with tidyverse and readxl
- Repository location: `1_code/1_0_ingest/1_0_0_SNAP_waiver_ingest.R`
- Inputs consumed: raw waiver panel workbooks stored under the input root
- Outputs produced: consolidated waiver data file under the processed root
- Invariant: date expansion must be deterministic, and the output must preserve the waiver records needed to reconstruct county-year treatment timing

Dependency 3: SNAP retailer ingest.

- Tool: R with tidyverse or data.table-compatible logic, but standardized to the new script structure
- Repository location: `1_code/1_0_ingest/1_0_1_SNAP_retailer_ingest.R`
- Inputs consumed: retailer locator raw file and county crosswalk
- Outputs produced: cleaned store-level SNAP data and county-year store count panel under the processed root
- Invariant: county FIPS formatting must be stable, and store-chain grouping must preserve the dollar-store categories used in the analysis

Dependency 4: ACS and unemployment inputs.

- Tool: R
- Repository location: `1_code/1_0_ingest/1_0_2_unemployment_rates.R`, `1_code/1_0_ingest/1_0_3_ACS_prep.R`
- Inputs consumed: the raw ACS and unemployment source files used in the legacy workflow
- Outputs produced: county-year covariate files used by the panel builder
- Invariant: year coverage and county identifiers must align with the retailer and waiver panels

Dependency 5: analysis-ready county-year panel.

- Tool: R
- Repository location: new script to be added under `1_code/1_0_ingest/`
- Inputs consumed: processed waiver panel, processed SNAP retailer outputs, ACS outputs, unemployment outputs, and any additional controls from the legacy U.S. script
- Outputs produced: one county-year analysis dataset for descriptives and reduced-form work
- Invariant: one row per county-year; stable event-time definitions; no silent dropping of treated counties

Dependency 6: descriptive and reduced-form scripts.

- Tool: R with `ggplot2` and `fixest`
- Repository location: new files under `1_code/1_1_descriptives/` and `1_code/1_2_reduced_form/`
- Inputs consumed: the analysis-ready panel and, where unavoidable, a small number of explicit auxiliary inputs such as rural-urban codes
- Outputs produced: figure files, tables, and model artifacts
- Invariant: these scripts should consume prepared data rather than repeating ingest logic
